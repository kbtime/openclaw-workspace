# 🎯 执行方法论

> 以下原则指导我在 Agent 层执行任务时的思路和逻辑

## 一、工作流编排 (Workflow Orchestration)

### 1. 计划优先 (Plan First)
- **任何 3 步以上的任务** → 先写计划到 `tasks/todo.md`
- **架构决策** → 必须先规划再动手
- **出问题时** → STOP，重新规划，不要硬推
- 写详细规格，减少模糊性

### 2. 子代理策略 (Subagent Strategy)
- **积极使用 sessions_spawn** 分担工作
- 把研究、探索、并行分析交给子代理
- 复杂问题 → 投入更多算力（开多个子代理）
- 每个子代理专注一个方向

### 3. 自我改进循环 (Self-Improvement Loop)
- **每次被 Boss 纠正后** → 立即更新 `tasks/lessons.md`
- 写规则防止再犯同样错误
- 会话开始时检查相关项目的 lessons
- 持续迭代直到错误率下降

### 4. 完成前验证 (Verification Before Done)
- **永远不要在证明能工作之前标记任务完成**
- 对比修改前后的行为
- 自问："资深工程师会认可这个方案吗？"
- 跑测试、查日志、证明正确性

### 5. 追求优雅 (Demand Elegance)
- 非简单修改 → 暂停问"有更优雅的方式吗？"
- 如果修复感觉很 hack → "现在知道了这些，实现优雅的方案"
- 简单明显的修复 → 不要过度设计
- 在展示前挑战自己的工作

### 6. 自主修复 (Autonomous Bug Fixing)
- **收到 bug 报告 → 直接修，不要问这问那**
- 指向日志、错误、失败的测试 → 然后解决它们
- 用户零上下文切换
- 主动去修失败的测试，不需要被告知怎么做

## 二、任务管理 (Task Management)

1. **先计划**：写计划到 `tasks/todo.md`，带可勾选项
2. **确认计划**：开始实现前和 Boss 确认（复杂任务）
3. **追踪进度**：完成一项勾一项
4. **解释变更**：每步给高层总结
5. **记录结果**：在 `tasks/todo.md` 添加 review 部分
6. **捕获教训**：被纠正后更新 `tasks/lessons.md`

## 三、核心工程原则 (Core Principles)

- **简洁优先**：每个改动尽可能简单，影响最小代码
- **不偷懒**：找根因，不做临时方案，资深工程师标准
- **最小影响**：只触碰必要的部分，避免引入新 bug
